# 性能预算与渲染优化方案

> 创建: 2026-06-07 | 状态: 方案 A 选定, 实施中

## 一、问题

游戏正常通关 20 波, wave 20 同时在场 200-220 怪. 无尽模式持续累加, 理论上不封顶.
问: 现在的 Canvas 2D 渲染能支持 1000 怪同时存在吗?

## 二、当前架构 (v1.3)

### Renderer (`src/engine/renderer.js`, 665 行)

纯 Canvas 2D, 无 GPU 加速, 无 sprite batching.

`drawEnemy` 每怪成本:
| 操作 | 调用次数 | 备注 |
|---|---|---|
| `ctx.save/restore` | 2 | 嵌套 save 浪费 |
| `globalAlpha` 状态切换 | 1 | 闪烁时 |
| `shadowColor/Blur` 状态切换 | **1 (常驻 12px)** | **头号瓶颈** |
| `beginPath + arc + fill` (主体) | 1 | 圆形怪 |
| `drawImage` (有图标) | 1 | 实际更省 |
| `fillRect` HP 条 (bg + fg) | **2** | 边框略 |
| `moveTo + lineTo + stroke` (方向) | 1 | 移动怪才画 |
| **总计** | **5-8 画/怪, 含 shadowBlur 慢路径** | |

**关键瓶颈**:
1. `ctx.shadowBlur = 12` 是 5-10x 慢于无阴影 (每像素多次重绘)
2. 无视口剔除 (offscreen 怪也画)
3. 无同色批处理 (每怪单独 set fillStyle)
4. 无对象池 (GC 压力)

| wave | spawn budget |

## 三、性能估算 (60fps = 16.67ms 预算)

| 怪数 | 画调用/帧 | 预计耗时 | 体验 |
|---|---|---|---|
| 200 (wave 20) | 1000-1600 | ~5-10ms | 流畅 |
| 400 | 2000-3200 | ~10-20ms | 30-50fps |
| 600 | 3000-4800 | ~15-30ms | 20-30fps 卡 |
| **1000** | 5000-8000 | **~30-50ms** | **20fps 死卡** |

**结论**: Canvas 2D + shadowBlur, 理论极限 500-600 怪 (40fps 临界). **1000 怪需要换底层**.

## 四、三个方案

### A. Canvas 2D 优化 (选定, 短期)
**目标**: 200 → 600 怪流畅

改动:
- 普通怪 `shadowBlur = 0` (只 Elite/Boss 留)
- 同色怪批处理 (按 color 分组, set fillStyle 一次画所有)
- 视口剔除 (offscreen 怪不画)
- HP 条: bg + fg = 2 画 (去掉边框)
- 嵌套 save/restore 合并
- 移动怪方向指示只在 isMovingEnemy 时画

代码量: 100-150 行改 renderer.js. 风险低. 0 依赖.

### B. 切换 PixiJS / Leafer (长期, 1000+ 怪)
**目标**: 2000-5000 怪

改动:
- 引入库: PixiJS (~200KB) 或 LeaferJS (~30KB)
- 重写 renderer.js 主体
- asset 改 sprite atlas
- 子弹/粒子/敌人全走 GPU
- 1-2 天工作

**Leafer vs PixiJS 对比**:

| | LeaferJS | PixiJS |
|---|---|---|
| 包大小 | ~30KB | ~200KB |
| 文档 | 中文 | 英文 |
| 矢量 | 强 | 一般 |
| **游戏 sprite batching** | **一般** | **强 (业界标准)** |
| 成熟度 | 新 | 2013 起, 行业标杆 |
| 适合场景 | UI 矢量 | **游戏专用** |

**推荐 PixiJS** (本项目是纯游戏).

### C. 先 A 后 B (推荐路径)
A 方案先落地, 满足 wave 20 + 短期无尽. 后续看需求 (online/massive wave) 再切 B.
